# Per-Chat MCP Session Filtering — Implementation Design

## Overview

Each CodeCompanion chat session gets its own MCP session on the combiner, identified by
a UUID token.  The combiner can apply a per-session server blocklist so a chat only sees
the MCP servers it is allowed to use.  This works identically for both ACP adapters
(opencode, Claude Code) and HTTP adapters (all other CC adapters: Copilot, Anthropic,
OpenAI, etc.).

---

## Concepts

### Token

A UUID v4 generated by the Lua plugin for each chat session, e.g.
`56b53afc-4e62-4e4d-9b8a-0f2af0bf7da8`.  It serves as the stable, durable identifier
the Lua side holds.  The combiner maps it to an internal MCP `session_id` once the remote
client connects.

### Session ID

The `mcp-session-id` HTTP header value assigned by FastMCP's StreamableHTTP transport
(hex string, no dashes, e.g. `b590612df488428ba501b32cd9382dbd`).  This is what the
combiner's filter tables are keyed on.  It is only known after the first HTTP request.

These are **two different IDs for the same session**:
- Token — stable, known ahead of time, used by Lua and the REST API
- Session ID — assigned dynamically by FastMCP, used internally for filter storage

### Pending Filter

If Lua POSTs a filter before the remote client has connected (common in the ACP path,
where opencode connects seconds after `session/new`), the combiner stores the filter as
*pending* keyed by token.  When the client connects and the session ID is assigned,
the pending filter is atomically applied.

---

## Singleton Client vs Per-Chat Clients

The plugin maintains a **singleton** MCP client that connects to `/mcp` (no token
suffix).  This is the global, session-agnostic connection used for:

- **Tool/resource/prompt discovery** — `tools/list`, `resources/list`, `prompts/list`
  are fetched once and cached.  CodeCompanion's tool registration UI needs the full
  unfiltered catalogue to render `@mcp` tool entries.
- **Health monitoring** — periodic `/health` polls and the SSE event stream that
  pushes server state changes to the Lua UI.
- **Fallback tool calls** — if a chat has no per-chat client (e.g. chat created before
  the combiner was ready), tool calls fall through to the singleton.

Because the singleton is **not** associated with any chat session, it sees **all**
servers and **no** per-session filter is applied.  This is intentional: it serves as
the control-plane view of the combiner.  Adding a token to this connection would
incorrectly imply session attachment and prevent it from showing the full catalogue.

Per-chat clients connect to `/mcp/<token>` and exist solely for filter-scoped
`call_tool` routing.  They use `lite=true` to skip capability fetching, SSE, and
polling — all of which the singleton already handles.

## Two Paths: ACP vs HTTP Adapter

```
CodeCompanion Chat
       │
       ├─── ACP adapter (opencode, claude-code)
       │         Token assigned in ACPSessionPre
       │         Combiner injected into session/new as /mcp/<token>
       │         opencode connects to combiner directly
       │         Lua tool calls go to CC → opencode (not to combiner)
       │
       └─── HTTP adapter (copilot, anthropic, openai, …)
                 Token assigned on CodeCompanionChatCreated
                 Lua creates lite per-chat MCP client at /mcp/<token>
                 All tool calls go Lua → per-chat client → combiner
```

Both paths use the same token-based filter REST API on the combiner side.

## ACP Path

### Sequence

```
CodeCompanionACPSessionPre fires
  → M._generate_token()             — UUID v4
  → M._pending_acp_tokens[adapter] = { token, agent_capabilities }
  → combiner entry injected into adapter_modified.defaults.mcpServers
      OR stored for patched transform_to_acp to pick up
  → stored on chat.adapter._mcp_token (so Post can read it)

Connection._establish_session() runs (immediately after Pre)
  → calls transform_to_acp() if mcpServers == "inherit_from_config"
  → our patched transform_to_acp appends the combiner entry:
      { type="http", name="mcp-combiner", url="http://host:port/mcp/<token>",
        headers=[{name="X-MCP-Combiner-Session", value=<token>}] }
  → sends session/new to opencode with the mcpServers list

CodeCompanionACPSessionPost fires
  → finds chat by matching acp_connection.session_id
  → reads token from chat.adapter._mcp_token
  → copies to chat._mcp_token
  → calls M._apply_token_filter(chat)

M._apply_token_filter
  → POST /sessions/token/<token>/filter
      { "allowed_servers": [...] }   or no-op if all servers allowed
  → combiner stores as pending (opencode hasn't connected yet)

opencode connects to http://host:port/mcp/<token>
  → TokenRewriteMiddleware rewrites /mcp/<token> → /mcp
  → FastMCP assigns mcp-session-id response header
  → TokenRewriteMiddleware: _token_sessions[token] = session_id
  → pending filter applied: _session_disabled[session_id] = pending_set
  → combiner sends notifications/tools/list_changed to session

opencode calls tools — combiner filters by session's disabled set
```

### mcpServers injection — two sub-cases

CodeCompanion ACP adapters declare `defaults.mcpServers` as one of:

| Value | Meaning | Who resolves it |
|-------|---------|-----------------|
| `"inherit_from_config"` | Inherit from global `config.mcp.servers` | CC calls `transform_to_acp()` inside `_establish_session` |
| `{}` (empty table) | No MCP servers by default; user/plugin adds them | `_establish_session` sends the table as-is |
| `{ ... }` (table with entries) | User-configured servers | `_establish_session` sends the table as-is |

mcp-companion handles each case:

#### Case 1: `"inherit_from_config"` (e.g. Copilot ACP)

`_establish_session` detects the string and calls `transform_to_acp()`.  Our patch
wraps this function (once, guarded by `_mcp_companion_patched`) to:

1. Call the original — it translates stdio-type servers from `config.mcp.servers` that
   are in `config.mcp.opts.default_servers`.
2. Also translate HTTP-type servers from `config.mcp.servers` that upstream ignores
   (upstream has no branch for `cfg.url`-based servers, only `cfg.cmd`).
3. Append the combiner entry using the pending token from `M._pending_acp_tokens`.

The combiner entry uses `/mcp/<token>` as the URL so the combiner can correlate the
connection to this specific ACP session.

> **Upstream limitation:** `transform_to_acp()` is called with no arguments, so
> `adapter_name` is nil.  The patch grabs the first (only) pending token entry —
> only one ACP session establishes at a time within a single Neovim instance.

#### Case 2: Concrete table — `{}` or `{ ... }` (e.g. OpenCode)

`transform_to_acp` is never called.  `ACPSessionPre` detects
`type(defaults.mcpServers) == "table"` and injects the combiner entry directly:

```lua
if defaults and type(defaults.mcpServers) == "table" then
  -- check idempotency, then:
  table.insert(defaults.mcpServers, combiner_entry)
end
```

This runs before `_establish_session` reads the table, so the combiner appears
in the `session/new` payload alongside any user-configured servers.

#### Idempotency

Both paths check for an existing entry with `name == "mcp-combiner"` before
inserting.  Re-establishing a session (e.g. after a disconnect/reconnect)
never duplicates the combiner entry.

### Why token is stored on `chat.adapter`

`ACPSessionPre` fires with `adapter_modified` in `args.data`.  At that point the chat
is found by matching `adapter.name`.  The token is stored on `adapter._mcp_token`.

`ACPSessionPost` fires with `session_id` in `args.data`.  The chat is found by matching
`acp_connection.session_id`.  The adapter reference on the chat object is the same
object as `adapter_modified` (CC does not copy it), so `chat.adapter._mcp_token` is
readable in Post.

## HTTP Adapter Path

### Sequence

```
CodeCompanionChatCreated fires
  → M._auto_http_tools(event_data)
  → adapter_name not in M._acp_adapter_names — proceed
  → M._setup_http_per_chat(chat)

M._setup_http_per_chat(chat)
  → M._generate_token()
  → chat._mcp_token = token
  → combiner.new_per_chat_client(token)     — lite client at /mcp/<token>
  → client:connect(callback)              — async TCP, initializes MCP session
    → on success:
        M._apply_token_filter(chat)       — POST filter immediately
    → on failure:
        chat._mcp_client = nil            — tool calls fall back to singleton

Tool call (any chat submit)
  → cc/tools.lua: _make_combiner_cmd resolves client
      client = _self.chat._mcp_client or singleton_client
  → call_tool routes through the per-chat client
  → combiner sees mcp-session-id header from the per-chat session
  → filter applied at combiner

CodeCompanionChatClosed fires
  → M._cleanup_session_filter(chat)
  → chat._mcp_client:disconnect()
  → DELETE /sessions/token/<token>/filter
```

### Per-chat client — lite mode

Full `connect()` on the singleton client:
- tools/list, resources/list, prompts/list
- health check
- starts SSE stream (persistent connection)
- starts polling timer

Per-chat clients don't need any of this — they only need an MCP session for
`call_tool` routing.  `lite=true` exits `connect()` after `initialize` +
`notifications/initialized`.  This avoids duplicate `combiner_ready` events,
unnecessary HTTP connections, and redundant capability fetches.

### ACP adapter detection timing

`CodeCompanionChatCreated` fires **before** `ACPSessionPre`.  So at `ChatCreated`
time we don't yet know if the chat is ACP-backed.

Solution: `M._acp_adapter_names` tracks adapter names seen in `ACPSessionPre`.
- For the **first** ACP chat on a given adapter: `_auto_http_tools` briefly creates a
  per-chat HTTP client.  `ACPSessionPre` fires shortly after and discards it
  (`c._mcp_client:disconnect()`).
- For **subsequent** ACP chats: `adapter_name` is already in `_acp_adapter_names`,
  so `_auto_http_tools` skips HTTP client setup entirely.

## Combiner: TokenRewriteMiddleware

`TokenRewriteMiddleware` (in `combiner/mcp_combiner/__main__.py`) is the outermost
middleware in the ASGI stack.  It runs before `MCPRequestLogMiddleware` and FastMCP.

**Input sources for the token (in priority order):**
1. URL path: `/mcp/<uuid>[/...]` — token extracted with regex, path rewritten to `/mcp`
2. HTTP header: `X-MCP-Combiner-Session: <uuid>` — fallback

**On first request for a new token:**
- Logs `Token not yet mapped`
- Calls `call_next(request)` (FastMCP processes it, sends back `mcp-session-id` header)
- Reads `mcp-session-id` from response headers
- Records `_token_sessions[token] = session_id`
- If a pending filter exists: atomically moves it to `_session_disabled[session_id]`

**On subsequent requests:**
- Logs at DEBUG `Token already mapped`
- Passes through without modification

The rewrite (`/mcp/<token>` → `/mcp`) happens in-place via `request.scope["path"]`
mutation.  This is safe because BaseHTTPMiddleware passes the same scope dict to
`call_next` — FastMCP receives the rewritten path and never sees the token in the URL.

---

## Combiner: Filter State

```python
_session_disabled: dict[str, set[str]]      # session_id → disabled server names
_token_sessions:   dict[str, str]           # token → session_id
_pending_token_filters: dict[str, set[str]] # token → disabled server names (pre-connect)
```

Filter enforcement happens in `_filter_tools()` (called from the `tools/list` handler).
Each tool is checked against the session's disabled set.  Disabled servers have all their
tools removed from the response.  When a filter changes, `notifications/tools/list_changed`
is sent to the affected session so the client refreshes its tool list.

## Combiner: REST API

### Token-based endpoints (used by Lua)

| Method | Path | Description |
|--------|------|-------------|
| `GET`  | `/sessions/token/<token>/filter` | Get current or pending filter state |
| `POST` | `/sessions/token/<token>/filter` | Set filter; stores as pending if not yet connected |
| `DELETE` | `/sessions/token/<token>/filter` | Clear filter and any pending state |
| `GET`  | `/sessions/token/<token>` | Resolve token → session_id |

### Session-ID-based endpoints (legacy / direct access)

| Method | Path | Description |
|--------|------|-------------|
| `GET`  | `/sessions` | List all active sessions |
| `GET`  | `/sessions/<session_id>/filter` | Get filter for a session |
| `POST` | `/sessions/<session_id>/filter` | Set filter for a session |
| `DELETE` | `/sessions/<session_id>/filter` | Clear filter for a session |

### POST body formats (both endpoints accept the same body)

```json
{ "allowed_servers": ["github", "filesystem"] }  // allow list — all others disabled
{ "disabled_servers": ["jira", "slack"] }         // explicit disable list
{ "enable": "github" }                            // re-enable a single server
{ "disable": "jira" }                             // disable a single server
```

`allowed_servers: []` means **disable all** (not "allow all").  The `_combiner`
meta-server is never included in the disabled set even if not in `allowed_servers`.

## Lua: Token Flow

```
chat._mcp_token         — UUID, set in ACPSessionPre or _setup_http_per_chat
chat._mcp_allowed_servers — nil (all) or string[] (allowlist)
chat._mcp_client        — per-chat MCP client (HTTP adapter only; nil for ACP)
chat.adapter._mcp_token — copy for ACPSessionPost to read (ACP only)
```

The token is the only persistent identifier Lua needs.  It never needs to know the
combiner's internal `session_id`.

---

## Lua: Tool Call Routing

In `cc/tools.lua`, `_make_combiner_cmd` is the factory for all MCP tool call commands.
At call time (not registration time) it resolves which client to use:

```lua
local client = _self.chat._mcp_client or singleton_client
```

- `_self` is the `CodeCompanion.Tools` orchestrator passed as the first argument to
  every tool `cmds[i]` function (from `runner.lua`).
- `_self.chat` is the active CC chat.
- If the chat has a per-chat client (`_mcp_client`), tool calls go through it.
- If not (combiner not connected, connect failed, or ACP chat), falls back to the
  singleton client.

The singleton client is captured at registration time and passed as `singleton_client`.

---

## `/mcp-session` Slash Command

The `/mcp-session` slash command (registered by `cc/session_commands.lua`) lets users
toggle individual MCP servers on/off for the current chat.

It uses the token endpoint for **both** ACP and HTTP adapter chats:
```
POST /sessions/token/<chat._mcp_token>/filter
  { "enable": "server_name" }   or   { "disable": "server_name" }
```

This unified path replaces the previous split implementation that used different
endpoints for ACP vs HTTP.  `chat._mcp_token` is set for all adapter types.

## Two Session ID Formats in Logs

When reading combiner logs you will see two different ID formats for the same session:

| Format | Example | Source |
|--------|---------|--------|
| UUID (dashes) | `9feacb93-e677-4b0c-94df-daeca9391b0b` | FastMCP's internal Python session object ID, logged by `MCPRequestLogMiddleware` via `context.fastmcp_context.session_id` |
| Hex (no dashes) | `b590612df488428ba501b32cd9382dbd` | `mcp-session-id` HTTP response header assigned by FastMCP's StreamableHTTP transport |

They identify the same session from different layers.  The filter enforcement in
`on_list_tools` and `on_call_tool` uses `context.fastmcp_context.session_id`, which
matches the value stored by `TokenRewriteMiddleware` from the `mcp-session-id` response
header.  The UUID format appears only in some log lines and is not used for filtering.

## Known Limitations / Future Work

- `transform_to_acp` (upstream CC) has no HTTP server branch and no nil guard on
  `cfg.cmd`.  Our patch adds HTTP support but the upstream function should be fixed
  (see `docs/todo.md`).
- The first ACP chat on a given adapter briefly creates then discards an HTTP per-chat
  client.  This is harmless but wasteful; a cleaner solution would require CC to fire
  `ACPSessionPre` before `ChatCreated` or expose the adapter type at `ChatCreated` time.
- Pending filters are held in memory.  If the combiner restarts between `ACPSessionPre`
  and opencode's first connect, the filter is lost (opencode connects with all servers
  enabled).
- Per-chat client `lite` mode does not fetch capabilities.  If tool registration ever
  needs to happen per-chat this will need to be revisited.
